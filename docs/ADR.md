# Architecture Decision Records

Решения, принятые на старте проекта. Формат: что выбрали, почему, какие альтернативы отклонили.

## ADR-1: SwiftUI, не Electron

**Статус:** Принято  
**Дата:** 2026-07-10

**Контекст:** Нужен нативный macOS-клиент. Выбор между SwiftUI и Electron.

**Решение:** SwiftUI.

**Причины:**
- Нативный look & feel macOS — Electron-приложения выглядят чужеродно
- Apple Silicon — нативная производительность, низкое потребление памяти
- Интеграция с системой: горячие клавиши, меню-бар, Keychain для ключей
- Существующий референс: `dodo-reach/hermes-desktop` (2k звёзд) доказал жизнеспособность SwiftUI-подхода

**Отклонённые альтернативы:**
- Electron (Hermes One, 13k звёзд) — тяжеловесный, не-нативный UI
- Tauri — лучше Electron, но всё ещё WebView, не полностью нативный
- Кастомный веб-интерфейс — не даёт macOS-интеграции

## ADR-2: API-подключение, не SSH

**Статус:** Принято  
**Дата:** 2026-07-10

**Контекст:** Как клиент общается с Hermes на VPS.

**Решение:** HTTP API (порт 8642) с API-ключом.

**Причины:**
- Hermes API уже работает — отдаёт сессии, стримит ответы
- Проще в реализации чем SSH: URLSession + SSE streaming
- Безопаснее: не нужен SSH-доступ, только API-ключ
- Не требует `hermes` CLI на клиенте
- `dodo-reach/hermes-desktop` использует SSH — и это его главное ограничение

**Отклонённые альтернативы:**
- SSH — требует полный shell-доступ, нет streaming API, сложнее обрабатывать ответы

## ADR-3: MVVM, не VIPER/TCA

**Статус:** Принято  
**Дата:** 2026-07-10

**Контекст:** Архитектурный паттерн для SwiftUI-приложения.

**Решение:** MVVM + Use Cases.

**Причины:**
- Достаточно для масштаба pet-проекта (5-7 экранов)
- Нативно для SwiftUI: `@Observable` ViewModel'и, `@Environment` для DI
- Не переусложняем: VIPER избыточен, TCA (The Composable Architecture) требует крутой кривой обучения

**Отклонённые альтернативы:**
- VIPER — слишком много слоёв для проекта такого размера
- TCA — мощно, но оверхед для pet-проекта

## ADR-4: Локальный SwiftData для топиков

**Статус:** Принято  
**Дата:** 2026-07-10

**Контекст:** Как хранить топики, настройки, историю на клиенте.

**Решение:** SwiftData (поверх CoreData/SQLite).

**Причины:**
- Нативно для SwiftUI: `@Query`, автоматическая реактивность
- Офлайн-доступ: топики и история доступны без API
- Не зависит от состояния сервера

## ADR-5: Feature-based структура папок

**Статус:** Принято  
**Дата:** 2026-07-10

**Контекст:** Как организовать файлы в Xcode-проекте.

**Решение:** Feature-based — каждый экран в своей папке со своими View, ViewModel, Models.

**Причины:**
- Удалить папку = удалить фичу
- Скопировать папку = переиспользовать фичу
- Изолированные модули = меньше конфликтов при параллельной работе
- Индустриальный стандарт для SwiftUI (см. статью Jerry PM, 2026)

**Структура:**
```
HermesDesktop/
├── App/              ← @main, DI-контейнер
├── Features/
│   ├── Chat/         ← основной чат
│   ├── Topics/       ← сайдбар с топиками
│   ├── Agents/       ← мониторинг subagent'ов
│   └── Settings/     ← настройки
├── Core/
│   ├── API/          ← HermesAPIClient, SSE streaming
│   └── Auth/         ← API-ключ, Keychain
├── Models/           ← доменные модели
├── DesignSystem/     ← цвета, шрифты, компоненты
└── Shared/           ← расширения, утилиты
```
