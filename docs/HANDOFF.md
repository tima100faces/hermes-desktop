## Где мы сейчас

Всё смержено в main и запушено в origin. За сессию 2026-07-10/11 сделано:

- **Единая модель Chat.** Сущности Project/Topic упразднены: остался один
  Chat с флагом `isPinned`. Секции сайдбара: «Pinned» (видна только если
  есть закреплённые) и «Chats» (единственная кнопка «+»). Старые данные
  мигрированы автоматически; прежние store-файлы сохранены как
  `.legacy-backup`, ничего не удалялось.
- **Два транспорта за одной абстракцией (ConversationService):**
  - Новые чаты — Sessions API (`/api/sessions/*`), стриминг через
    `POST /api/sessions/{id}/chat/stream` (события: `assistant.delta`,
    `tool.started` — превью в поле `preview`, `run.completed` несёт
    финальный результат; `tool.completed` результата не несёт).
  - Закреплённые чаты (бывшие темы) — старый Runs API (`/v1/runs`,
    события `message.delta`, `tool.call`, `tool.result`, `run.completed`).
  - UI не знает, каким API обслуживается разговор.
- **Список чатов — только локальный.** Никогда не строить его из общего
  списка сессий сервера без фильтра `source`: в той же базе живут сессии
  из Telegram — их не показывать, не переименовывать, не удалять.
- **Интерфейс полностью на английском** (правило в CLAUDE.md). Документация
  и общение с Тимом — по-русски.
- **Починен SSE-парсер (SSEClient):** `bytes.lines` на этой macOS глотает
  пустые строки → события склеивались, выживало только последнее. Теперь
  парсер сам режет сырые байты на строки и собирает события по спецификации
  SSE (граница — пустая строка, несколько `data:` склеиваются через `\n`).
  Полный прогон тестов зелёный.
- **SSE-парсинг вынесен в общий компонент (`SSEFrameParser`).** SSEClient и
  SessionSSEClient — теперь тонкие обёртки над ним, каждая знает только свои
  имена событий. Заодно в SessionSSEClient нашёлся и попутно починился
  второй баг, отдельный от «глотания» пустых строк: несколько `data:`-строк
  внутри одного события не склеивались через `\n`, а перезаписывали друг
  друга (выживала только последняя) — из-за этого у Sessions API ломались
  события с многострочной `data:`, даже если бы границы по пустой строке
  сработали правильно. Один полный набор тестов на `SSEFrameParser` +
  маленький набор на каждую обёртку (SSEClientTests — прежний, без
  изменений; SessionSSEClientTests — новый). Полный прогон юнит-тестов
  зелёный (92/92).
- **Троттлинг отрисовки стриминга:** дельты копятся в
  `pendingStreamingContent` без задержки, видимый `streamingContent`
  обновляется не чаще ~70 мс; `run.completed`/stop публикуют хвост сразу.
  Автоскролл развязан со стримингом через `streamingUpdateTick`.
- Имя агента больше не захардкожено: настраивается в Settings, пустое
  значение показывает «Hermes».

## Беклог — по приоритету

1. **Этап 3 — «Проекты»** (главная следующая фича): группа чатов с общими
   инструкциями и общей памятью. Продуктовое решение принято, техразведка
   перепроверена вживую 2026-07-11 (создавались и сразу удалялись тестовые
   сессии на проде):
   - **Инструкции подтверждены.** Поле `instructions` в теле
     `POST /api/sessions/{id}/chat` и `/chat/stream` реально накладывает
     системное сообщение поверх личности агента (сессия получает
     `has_system_prompt: true`, агент дословно следует инструкции).
     Надёжно срабатывает, если отправлено в новую сессию; в сессии с уже
     идущей историей может не сработать сразу — обычное поведение модели,
     не баг транспорта, но стоит учесть в UX (эффект от смены инструкций
     на идущий чат может быть не мгновенным).
   - **Общая память — НЕ через заголовок.** `X-Hermes-Session-Key`
     сервер принимает и эхо-возвращает на обоих эндпоинтах, но НЕ даёт
     изоляции памяти: факт, сохранённый агентом под одним ключом, тут же
     доступен из сессий с другим ключом и вовсе без заголовка. Причина —
     мы используем встроенную файловую память через git-синк (agents-hub),
     провайдера с поддержкой скоупов нет, поэтому ключ сейчас без эффекта
     — это ожидаемо. Решение: всё равно генерировать и слать ключ на
     проект (на будущее, вдруг Hermes научится его использовать), но не
     заявлять изоляцию памяти как фичу — долгосрочная память Ржавчика в
     любом случае одна на все чаты, в проектах и вне них.
   - Транспорт для чатов проекта — Sessions API (как у обычных чатов
     сейчас).
   Далее — план реализации (часть 2 задачи), НЕ код сразу.
2. **Миграция закреплённых чатов с Runs API на Sessions API.** Известное
   препятствие: у серверных сессий нет `conversationKey` — прямого
   сопоставления «локальная тема ↔ серверная сессия» нет (проверено
   экспериментом). Отдельная задача, требует обсуждения подхода.
3. Автоскролл при ручной прокрутке вверх во время ответа — не утаскивать
   вниз против воли (UI-SPEC §10).
4. Инкрементальный парсинг в MarkdownRenderer (сейчас полный перепарсинг
   всего текста на каждый тик).
5. Текст результата инструмента в карточках новых чатов — ограничение
   Sessions API (`tool.completed` без результата), следить за апдейтами
   Hermes.
6. Retry в hover-ряду — под вопросом, не браться без явной просьбы
   (спека сохранена в истории, владелец не уверен в нужности).

## Правила (коротко; полностью — CLAUDE.md и docs/UI-SPEC.md)

- Hermes — внешний сервис: код не трогаем никогда, API используем
  полноценно; конфиг/версия/SSH на VPS — только по явному разрешению
  Тима в текущей сессии.
- Перед началом любой задачи — описать план словами и дождаться «ок».
- Никогда не снимать скриншоты экрана пользователя; визуальная проверка —
  всегда за Тимом.
- Работа в feature-ветках; в main — после подтверждения Тима; после мержа
  в main — пушить в origin сразу.
- Строки интерфейса — только английские; документация и общение — русские.
- Владелец — дизайнер, не программист: объяснять просто, без жаргона.
