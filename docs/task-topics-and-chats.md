# Задача: Темы и свободные чаты (этапы 0–2)

Перед началом работы обязательно прочитай `CLAUDE.md` и `docs/UI-SPEC.md`.
Работа ведётся по принятому в репозитории git-workflow: feature-ветка,
в main — только после того, как Тим собрал билд и подтвердил.

## Контекст

Сейчас в приложении есть сущности «Проекты», каждая из которых на деле —
одна непрерывная сессия Hermes с постоянным `conversationKey`. Это
семантически соответствует **топикам** Hermes (как топики в Telegram DM),
а не проектам в смысле Claude. Мы приводим терминологию в порядок и
добавляем свободные чаты — отдельные разовые сессии, как новые чаты
в приложении Claude.

Проекты в стиле Claude (группа чатов с общими инструкциями) — это
**этап 3, в эту задачу НЕ входит**. Не реализовывать, не закладывать
преждевременные абстракции под него.

## Этап 0 — разведка (обязательно первым, до любого кода)

1. Запроси `GET /v1/capabilities` у Hermes-сервера пользователя
   (URL и ключ — те же, что использует приложение; ключ в Keychain,
   auth — `Authorization: Bearer <API_SERVER_KEY>`).
2. Проверь наличие флагов `session_*` и записей `endpoints.session_*`.
   Нужные эндпоинты Sessions API (все под `/api/sessions/*`):
   - `GET /api/sessions` — список (пагинация: limit, offset)
   - `POST /api/sessions` — создать пустую сессию
   - `GET /api/sessions/{id}` — метаданные
   - `PATCH /api/sessions/{id}` — переименование (title)
   - `DELETE /api/sessions/{id}` — удаление
   - `GET /api/sessions/{id}/messages` — история сообщений
   - `POST /api/sessions/{id}/chat/stream` — один ход агента по SSE
     (события: `assistant.delta`, `tool.started`, `tool.completed`,
     `run.completed`)
3. Изучи текущий сетевой слой клиента: каким эндпоинтом приложение
   общается с Hermes сейчас, как устроен стриминг, где живёт
   `conversationKey`.
4. **Останови работу и отчитайся Тиму** коротким резюме: какая версия
   Hermes на сервере, какие session-флаги есть, чем клиент пользуется
   сейчас, и вывод — можно ли переходить на Sessions API или сначала
   нужно обновить Hermes на VPS. Дальше двигайся только после его «ок».

## Этап 1 — переименование «Проекты» → «Темы»

- Все строки интерфейса: «Проекты» → «Темы», «Select a project» →
  «Выбери тему», и т.д. по всему UI.
- Переименуй модель `Project` → `Topic` и связанные имена
  (viewmodel, файлы, комментарии). Данных у пользователя немного,
  но молча терять их нельзя: предложи Тиму вариант миграции
  (лёгкая миграция SwiftData или одноразовый перенос) до реализации.
- Обнови `docs/UI-SPEC.md` и `README.md` в тех же коммитах.

## Этап 2 — свободные чаты

### Модель и данные
- Единая сущность разговора с полем вида `kind: chat | topic`
  (или отдельная сущность — реши по месту, но без дублирования логики).
- У чата: серверный session id, локальный заголовок, дата последней
  активности.

### Сайдбар
- Две секции: **Чаты** (сверху) и **Темы**. Стиль — строго по
  UI-SPEC (ScrollView + кнопки, никаких List; цвета только Color.hk*).
- Кнопка «Новый чат» → `POST /api/sessions`, открытие пустого чата.
- Чаты сортируются по последней активности, темы — как сейчас.
- Контекстное меню у чата: переименовать (`PATCH`), удалить
  (`DELETE`, с подтверждением).

### Поведение чата
- Отправка сообщения — через `POST /api/sessions/{id}/chat/stream`,
  рендер стриминга и статусов инструментов как в существующих темах
  (`tool.started`/`tool.completed` маппятся на текущие карточки
  статусов subagent'ов, `assistant.delta` — на стриминг текста).
- Автозаголовок: после первого сообщения — первые ~40 символов
  текста пользователя, отправить на сервер через `PATCH` (title),
  чтобы заголовок был единым локально и на сервере.
- При открытии чата история подтягивается с сервера
  (`GET /api/sessions/{id}/messages`), локальный кэш — вторичен.

### Не ломать
- Существующие темы должны продолжить работать без изменений
  для пользователя — тот же ключ, та же история.
- Empty state: если нет ни чатов, ни тем — предложение создать
  первый чат.
